Data Security Principles/Patterns

JWT Authentication & Authorization

JWT Authentication: I use JSON Web Tokens (JWT) to authenticate users is a key security principle. When a user logs in, a JWT token is created and sent to the client. This token is then used in subsequent requests to authenticate the user by verifying the token on the server. The authenticateToken middleware checks for the presence of the token in the request headers and verifies its authenticity using a secret key

Role-Based Authorization

After the JWT token is verified, user roles (such as admin or patient) are used to determine access to various resources. This ensures that only users with appropriate roles can access specific pages or data (e.g., only admins can access the /admin page).

Middleware Pattern

I use middleware for authentication. (authenticateToken) is a common pattern in Express.js applications. Middleware functions provide a way to centralize logic like authentication and authorization checks, ensuring that they are executed before routing to specific endpoints..

Database Querying & Security

I use SQL Queries to fetch and insert data from a relational database. The queries are parameterized, meaning that user inputs (e.g., email, password) are passed as parameters to avoid SQL injection attacks. For example, in the login section, the db.query method uses ? to safely inject parameters into SQL queries

Password Hashing

In the registration route, passwords are hashed using bcrypt before being stored in the database, which is a security best practice for protecting sensitive data like passwords

Session Management

Session Handling:My application uses sessions to manage logged-in users. For example, when a patient or admin logs in, their ID is stored in the session (req.session.patientId or req.session.adminId). This session-based approach provides a way to keep track of the user's login status across requests..

Serving Static HTML Pages Dynamically

In my application,i dynamically serve HTML pages (e.g., index.html, admin.html, login.html) by reading them from the file system and sending them to the client. This approach allows for flexibility in serving different pages based on user roles or actions

Error Handling

I include code for error handling for various operations, such as reading files (fs.readFile), querying the database, and handling errors during password hashing. Proper error handling ensures that users receive meaningful error messages and prevents unhandled exceptions from crashing the server..

Handling Role-Specific Routes and Access Control

I use role-based access control. (RBAC) is implemented, where routes like /admin are protected by JWT authentication and role checks to ensure that only users with the admin role can access them. If a user doesn't have the required role, a 403 Forbidden status is returned..

Session-Based Conditional Rendering

I Implement logic for rendering pages conditionally based on the user's session (req.session.patientId or req.session.adminId) ensures that only authenticated users can access certain pages (like /profile, /doctor). If the user is not authenticated, they are redirected to the login page..

Secure Communication (HTTPS)

I use SSL/TLS certificates (server.key and server.cert) indicates that HTTPS is implemented, ensuring secure communication between the client and the server. This is crucial for protecting sensitive data, such as passwords, during transmission..

Redirect Logic

I include logic to redirect users to appropriate pages based on certain conditions, such as after successful login or registration (res.redirect('/login') or res.redirect('/admin.html')). This provides a smoother user experience..
